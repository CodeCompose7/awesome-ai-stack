# 릴리스 체크리스트

새 버전을 낼 때 갱신·수행할 것들을 모은 문서입니다. 순서대로 따라가면 됩니다.
마크다운 작성 규칙은 [markdown-style.md](markdown-style.md)를 따릅니다 — 특히
한국어 문장을 `).`로 끝내지 않도록 주의합니다.

## 버전 정하기

[유의적 버전(SemVer)](https://semver.org/lang/ko/)을 따릅니다.

- **PATCH**(`x.y.Z`) — 버그 수정, 문구·스타일 손질처럼 기능이 늘지 않는 변경
- **MINOR**(`x.Y.0`) — 새 도구·개념·글·샘플, 새 페이지나 기능처럼 하위 호환이 유지되는 추가
- **MAJOR**(`X.0.0`) — 라우트 구조나 콘텐츠 스키마가 깨지는 변경

## 갱신할 것

| 무엇 | 파일 | 비고 |
| --- | --- | --- |
| 버전 번호 | [`package.json`](../package.json) | `"version"` 필드를 새 버전으로 |
| 변경 이력 | [`CHANGELOG.md`](../CHANGELOG.md) | 맨 위에 새 섹션 추가(아래 형식 참고) |

버전 문자열은 이 두 곳에만 있습니다. 테마(`stack-site-builder`)는 자기
저장소에서 따로 버전을 관리하므로 이 사이트의 릴리스와는 무관합니다.

## CHANGELOG 작성 형식

기존 항목과 같은 형식을 지킵니다 — 파일 위쪽 항목들을 그대로 참고하면 됩니다.

1. 맨 위(안내 문단 바로 아래)에 `## X.Y.Z - YYYY-MM-DD` 헤더를 추가
2. 그 아래 이번 릴리스의 핵심을 요약하는 짧은 문단을 한 편 씀
3. 변경을 `### 추가됨 X.Y.Z` · `### 변경됨 X.Y.Z` · `### 수정됨 X.Y.Z`로 나눠 적음 —
   해당 없는 구분은 생략하고, 각 헤더 뒤에 버전 번호를 붙이는 점에 주의
4. 각 항목은 **굵은 요약** — 설명 꼴로 씀. 명사로 끝나는 항목에는 마침표를 붙이지 않음

지난 커밋을 훑어 초안을 잡으면 편합니다.

```bash
git log --oneline vX.Y.Z_이전..HEAD   # 지난 태그 이후의 커밋 목록
```

## 마무리 확인

릴리스 커밋 전에 빌드가 깨끗한지 확인합니다.

```bash
pnpm check   # 타입·Astro 진단 (오류 0이어야 함)
pnpm build   # 전체 사이트 빌드 성공
```

## 릴리스 커밋과 배포

이 저장소는 git-flow(no-ff 병합)를 씁니다. 배포는 `main`에 푸시될 때
[`.github/workflows/deploy.yml`](../.github/workflows/deploy.yml)가 GitHub Pages로
올립니다.

1. 릴리스 브랜치에서 위 두 파일만 담아 릴리스 커밋을 만듦

   ```bash
   git commit -am "chore(release): X.Y.Z"
   ```

2. `main`에 병합해 배포를 트리거하고, `develop`에도 되돌려 병합
3. 태그를 만들고 푸시

   ```bash
   git tag vX.Y.Z
   git push --tags
   ```

> GitHub의 별 수·최신 버전은 빌드 시점에 가져오며, 배포 워크플로가 매일
> 다시 빌드하므로 코드 변경이 없어도 최신 상태로 유지됩니다.
